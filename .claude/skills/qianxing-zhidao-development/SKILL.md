## Mock 数据规范

- `image: string` — 占位图（CSS渐变或emoji模拟）

## AI 能力模拟规范

由于Demo阶段不接入大模型API，所有AI能力通过规则引擎模拟：

1. **画像生成：** 根据用户选择的兴趣标签，用加权规则生成游客画像
2. **路线推荐：** 基于画像与路线的标签匹配度进行打分排序
3. **AI导游问答：** 从知识库中匹配最相关的问答对，加随机延迟模拟思考
4. **文案风格：** AI生成文案应模仿"AIGC风格"，正式、有条理、有数据感

## UI 美化规范（Web + 小程序通用，强制执行）

### 动画：强制使用 GSAP

- Web 端：`import gsap from 'gsap'`（已安装），每个页面必须有入场动画
- 小程序端：等效实现（`wx.createAnimation` 或纯 JS 动画）
- 卡片列表：stagger 依次出现
- 页面切换：过渡动画
- 禁止 CSS `@keyframes` 作为主要动画手段

### 资源：图片和视频优先

- **优先级：真实图片/视频 > SVG 图标 > 纯文字**
- **禁止 Emoji 图标**（🧭🏞️🤖 等不允许出现在正式页面上）
- 功能图标使用 Element Plus Icons（Web）或 SVG（小程序）
- 景点展示必须使用真实照片
- 图片缺失时：CSS 渐变占位 + 立即向用户索要
- **找不到图就找用户要，不要自己用 Emoji 替代**

### 图片获取流程

```
需要图片 → 搜索 Unsplash/Pixabay/Pexels
         → 找到 → 下载到 assets 目录
         → 找不到 → 列出缺失清单向用户索要
```

### 色彩

- 主色：`#1f8f5f`（翠绿 · 贵州山水）
- 辅色：`#2f6bff`（科技蓝 · AI智能）
- 强调：`#e67e22`（暖橙 · 苗族元素）
- 背景：`#f4f8f5`（浅绿灰）
- 不引入第四种主题色

### 美观度标准

- 大留白 + 清晰信息层级，字重对比明显（标题 700+ / 正文 400）
- 移动端按钮高度 ≥ 44px（Apple HIG）
- 图片必须高清（Web ≥ 1920px，小程序 ≥ 750px）
- 毛玻璃卡片、渐变光影、圆角 12-30px
- 白色半透明底 + 细微边框 + 柔和阴影

## 每次开发流程（强制执行）

```
1. 输出开发计划
   - 本次目标（一句话）
   - 所属端（Web 端 / 小程序端 / 双端共享能力 + 划分理由）
   - 涉及文件（新建/修改列表）
   - 预期效果（用户可见）
   - 验证方式（如何确认成功）

2. 等待确认
   - 用户回复"确认"或"继续"后进入第3步

3. 编写代码
   - 先创建类型定义（如需要）
   - 再创建Mock数据（如需要）
   - 再创建组件/页面
   - 最后配置路由

4. 验证
   - 运行项目确认不报错
   - 确认页面可正常访问
   - 确认与已有页面无冲突
```

## 禁止事项

1. **禁止一次性大规模修改** — 每次只完成一个小步骤
2. **禁止破坏已运行的代码** — 每次修改后项目必须仍能正常运行
3. **禁止引入复杂依赖** — 不接入 ECharts、地图 SDK、真实 AI API、后端接口、数据库
4. **禁止占位符文案** — 所有文案正式、适合比赛展示
5. **禁止普通旅游网站风格** — 保持科技感 + 文旅感 + 贵州特色
6. **禁止混用命令语法** — Windows环境优先使用PowerShell
7. **禁止跳过类型定义** — 所有数据必须先有TypeScript类型
8. **未经确认不写代码** — 先输出计划，等待确认后再实施
9. **统计展示只用纯组件** — 使用 Element Plus 卡片、进度条、列表，禁止 ECharts

## 多端开发总原则

本项目采用"双端协同"策略：

- **小程序端：** 游客端核心产品，优先开发，负责真实用户使用场景。
- **Web 端：** 项目展示官网、后台管理端、数据统计看板、比赛路演展示端。
- 两端不是互相替代关系，而是共同组成完整项目。

### 小程序端定位

小程序端面向游客真实使用，优先完成：首页、游客兴趣选择、游客画像生成、个性化路线推荐、路线详情、AI 智能导游、景点知识库、景点详情、我的行程。

小程序端开发要求：使用微信原生小程序结构；优先适配手机端；页面交互简洁顺手适合游客边走边用；不接真实后端前使用本地 Mock 数据；适合录制比赛演示视频。

### Web 端定位

Web 端不删除，继续保留并逐步完善：项目展示官网、比赛路演展示页、后台管理端、数据统计看板、景点管理、路线管理、用户画像数据展示、AI 问答记录展示。

Web 端开发要求：不再作为游客端主入口；不删除已有 Web 页面；已有游客流程页面可保留为演示版；后续重点转向后台管理和路演展示。

### 双端共享原则

- Mock 数据可以各端单独存放，但字段设计要保持同步（景点、路线、兴趣标签、用户画像、AI 问答）
- 后续后端接口要同时服务 Web 和小程序
- 不要在 Web 端和小程序端设计两套完全不同的业务模型

### 每次开发前必须判断

- 游客真实使用功能 → 优先放小程序端
- 后台管理、数据统计、路演展示 → 优先放 Web 端
- 两端都需要的能力 → 先统一数据结构和接口设计，再分别实现

### 禁止事项（补充）

1. 不要把小程序端做成 Web 后台样式
2. 不要把 Web 端完全删除
3. 不要让 Web 和小程序的数据字段各写各的
4. 不要为了复用代码强行把两端工程混在一起
5. 不要一边开发小程序，一边破坏 Web 端已有成果
6. 不要在没有确认的情况下引入 uni-app、Taro、Pinia、后端接口、地图 SDK 或真实 AI API
7. 不要忘记比赛演示目标：小程序展示游客体验，Web 展示项目能力和后台价值
