# 黔行智导：贵州山地旅游 AI 伴游与安全守护平台

## 项目背景

- **赛事：** 2026年贵州省人工智能创业大赛
- **赛道：** 行业应用赛道 · 赛题三：围绕山地旅游游客兴趣建模与个性化导览
- **团队：** Maxwell
- **阶段：** 第二阶段 — 小程序体验深化 + 行程功能产品化 + 双端后端架构准备

## 项目定位

面向贵州山地旅游、智慧景区、乡村文旅、旅行社、民宿集群和地方文旅数字化管理场景，
基于大语言模型、RAG知识库、游客兴趣画像、个性化推荐算法、动态路径规划和AI伴游问答能力，
为游客提供个性化贵州旅游路线推荐、景点导览、智能问答、安全守护和出行辅助服务。

核心差异：不只是旅游推荐，更是"AI 伴游 + 安全守护"——覆盖行前规划、行中伴游、行中安全三条主线。

## 二、赛题对齐原则

本项目必须严格围绕赛题三：

**围绕山地旅游游客兴趣建模与个性化导览**

不能只做普通旅游推荐，必须逐步体现赛题中的四个核心方向：

### 方向一：游客兴趣建模与个性化导览

识别游客兴趣、出行人群、体力偏好、游览节奏，生成个性化路线、讲解内容、小众打卡推荐。

- 当前阶段：规则算法打分（标签匹配 + 体力等级 + 人群适配 + 天数匹配）
- 后续阶段：ML模型 + 协同过滤 + 实时行为追踪

### 方向二：山地动态路径规划与实时重规划

在当前阶段先用规则算法模拟，考虑体力消耗、景点热度、天气变化、拥堵情况、游览时间，提供路线优化和替代方案。

- 当前阶段：规则模拟（体力衰减模型 + 景点耗时估算 + 替代路线推荐）
- 后续阶段：实时路况 + 天气API + 动态重规划算法

### 方向三：游客异常行为识别与风险预警

在当前阶段先用模拟数据展示，例如偏离路线、长时间停留、体力下降、天气突变、高风险区域提醒。

- 当前阶段：规则风险评分（停留超时检测 + 偏离路线检测 + 体力预警阈值）
- 后续阶段：实时GPS轨迹分析 + ML异常检测模型

### 方向四：游客求助与安全辅助

提供附近服务点、医疗点、救援点、安全提醒、紧急联系和当前位置辅助信息，体现"智能伴游 + 安全守护"。

- 当前阶段：预设服务点坐标 + 一键拨打紧急电话 + 安全贴士
- 后续阶段：实时定位 + 最近救援点导航 + SOS联动

## 三、评审导向原则

后续开发必须服务于以下评审要点：

### 1. 创新性

页面和功能要体现 AI、游客画像、智能推荐、动态重规划、安全预警，不要只做普通旅游网站。

### 2. 技术难度

即使当前使用本地数据，也要体现清晰算法逻辑、数据结构、风险评估规则、路线推荐评分机制。

### 3. 完成度

小程序游客端必须能走通完整流程：

```
首页 → 兴趣选择 → 画像生成 → 路线推荐 → 路线详情 → AI伴游问答 → 安全提醒 → 收藏/保存行程
```

### 4. 应用价值

功能必须贴近贵州山地旅游真实痛点：
- 路线规划难（景点分散、山路复杂）
- 体力分配难（山地海拔变化大）
- 安全风险高（天气多变、信号盲区）
- 民族文化讲解不足（苗族、侗族、布依族深度内容）

## 四、双端定位

本项目采用"双端协同"：

### 小程序端（游客主产品 · 优先开发）

小程序端是游客真实使用的主产品。功能包括：

1. 首页
2. 兴趣选择
3. 游客画像
4. 个性化路线推荐
5. 路线详情
6. AI 山地伴游
7. 安全守护
8. 景点知识库
9. 景点详情
10. 我的行程

小程序端要求：
- 微信原生小程序结构
- 移动优先，操作简单
- 适合游客在景区、路上、山地游览过程中使用
- 页面交互简洁顺手，按钮大、文字清晰
- 不接真实后端前，使用本地数据
- 所有文案面向真实游客

### Web 端（项目展示 + 运营管理）

Web 端负责：

1. 项目官网展示
2. 文旅运营管理
3. 景点管理
4. 路线管理
5. 游客画像数据洞察
6. AI 问答记录
7. 安全预警记录
8. 数据统计看板

Web 端要求：
- 不作为游客主入口
- 不删除已有 Web 页面
- 已有游客流程页面保留为 Web 体验版
- 重点转向后台管理和数据洞察
- 风格保持科技感、文旅感、贵州地域特色

### 双端共享原则

- 景点/路线/兴趣标签/用户画像/AI 问答/安全提醒 数据字段保持同步
- Mock 数据可各端单独存放，但字段设计保持一致
- 后续后端接口同时服务 Web 和小程序
- 不在 Web 端和小程序端设计两套完全不同的业务模型

## 七、小程序前端深化规则

页面美化必须提升用户完成任务的效率，不做无意义装饰。

### 布局有用原则

1. 每个视觉模块必须能回答"用户为什么需要它"；
2. 不为了好看增加无意义卡片；
3. 所有按钮必须有明确去向；
4. 不再新增临时入口、重复入口、无意义入口。

### 首页核心任务

首页要突出三件事：
- 开始规划
- AI助手
- 我的行程

### 路线详情核心任务

路线详情要突出：
- 行程安排
- 安全提醒
- 保存/查看行程
- AI伴游

### 通用约束

1. 我的行程要逐步成为用户复访核心入口；
2. AI助手要能承接路线、景点、知识库上下文；
3. 页面布局必须移动端优先，底部 tabBar 不得遮挡内容；
4. 新增页面必须考虑与 tabBar 的兼容性；
5. 页面美化必须服务真实使用场景，禁止纯装饰性改动。

## 九、后端架构准备规则

下一阶段开始逐步规划后端，但不要一上来乱接接口。

### 双端共用原则

后端必须同时服务：
1. Web 管理端
2. 微信小程序游客端

后端不是只给小程序用，也不是只给 Web 后台用。

### 后端核心模块规划

至少包括：

1. 景点管理
2. 路线管理
3. 兴趣标签管理
4. 游客画像
5. 路线推荐
6. 我的行程
7. AI伴游问答
8. 安全提醒
9. 服务点管理
10. Web 后台数据看板

### API 设计原则

1. 同一套业务数据，Web 和小程序共用；
2. Web 负责管理数据；
3. 小程序负责消费数据；
4. 不要为 Web 和小程序设计两套割裂模型；
5. 接口路径统一使用 `/api/...`；
6. 管理端接口使用 `/api/admin/...`；
7. 小程序游客端接口使用 `/api/app/...`；
8. 公共数据接口可以使用 `/api/public/...`；
9. 设计接口前必须先设计数据模型；
10. 后端开发前必须先输出表结构、接口清单、数据流图。

## 命名规范

- **视图文件：** `XxxView.vue`（PascalCase + View后缀）
- **组件文件：** `XxxYyy.vue`（PascalCase，按功能分组到子目录）
- **类型文件：** `xxx.ts`（camelCase，按领域拆分）
- **路由路径：** `/xxx` 或 `/xxx/:id`（kebab-case）
- **CSS类名：** `xxx-yyy`（kebab-case，BEM风格）
- **TypeScript接口：** 语义化命名（如 `Attraction`、`TourRoute`、`InterestTag`）
- **组合式函数：** `useXxx`（camelCase）

### UI 设计铁律（Web + 小程序通用）

**以下规则适用于本项目所有前端开发，违反即视为不合格。**

#### 1. 动画强制使用 GSAP

所有页面入场动画、滚动触发动画、过渡动效必须使用 GSAP（GreenSock Animation Platform）：

- Web 端：`import gsap from 'gsap'`，已安装
- 小程序端：使用小程序原生 `wx.createAnimation` 实现等效效果
- 禁止使用 CSS `@keyframes` 作为主要动画手段（GSAP 性能更好、更可控）
- 每个页面至少要有入场动画（元素从下往上淡入）
- 卡片列表使用 stagger 依次出现
- 页面切换时有过渡动画

#### 2. 图片和视频优先，禁止 Emoji 图标

**优先级：真实图片/视频 > SVG 图标 > 纯文字**

- 景点展示：必须使用真实景点照片，禁止 Emoji 图标代替
- 品牌/氛围：Hero 区优先使用贵州实景照片或视频背景
- 功能图标：使用 Element Plus Icons（Web）或 SVG（小程序），禁止 Emoji（🧭🏞️🤖❌）
- 图片缺失时：使用 CSS 渐变占位 + 文字标签，**并立即向用户索要图片**
- 用户原话："需要图你找不到就找我要"
- 每个页面的图片需求必须在开发计划中列出
- 找不到的图片：先搜索 Unsplash/Pixabay/Wikimedia，找不到就向用户报告缺失清单

#### 3. 美观度标准

- 大留白、清晰的信息层级
- 卡片与背景有明确的视觉分离
- 字重对比明显（标题 700+ vs 正文 400）
- 色彩克制：主色 `#1f8f5f` + 辅色 `#2f6bff` + 暖橙 `#e67e22`，不引入第四种主题色
- 移动端按钮高度 ≥ 44px（符合 Apple HIG 触摸标准）
- 图片必须高清（≥ 1920px 宽度，小程序可降至 750px）

#### 4. 图片获取流程

```
需要图片 → 先搜索 Unsplash / Pixabay / Pexels
         → 找到了 → 下载到对应 assets 目录
         → 找不到 → 列出缺失清单，向用户索要
         → 用户提供 → 放入 assets 目录
```

## 🧬 自动加载规则（每次会话自动生效，无需手动调用技能）

以下规则已从技能文件中提取并直接熔入此处，你每次写代码自动遵守：

### 不可变性（CRITICAL · 始终生效）
- 永远创建新对象，**绝不原地修改**已有对象
- 使用 spread `{ ...obj, field: newVal }` 做不可变更新
- 函数返回新值，不修改参数

### TypeScript 铁律（始终生效）
- 导出函数**必须**显式标注参数和返回类型，让 TS 推断局部变量
- **禁止 `any`** — 用 `unknown` + 安全 narrowing 替代
- 组件 Props: `defineProps<MyProps>()` + named `interface`
- `interface` → 对象形状，`type` → 联合/交叉/工具类型
- 优先 `string literal union`，不用 `enum`（除非互操作需要）

### 命名规范（始终生效）
| 类型 | 规范 | 示例 |
|------|------|------|
| 变量/函数/文件名 | `camelCase` | `getUserProfile` |
| Boolean | `is/has/should/can` 前缀 | `isLoading` |
| Interface/Type/Component | `PascalCase` | `UserCard` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_RETRY` |
| Composables | `use` + `PascalCase` | `useDebounce` |
| 视图文件 | `XxxView.vue` | `HomeView.vue` |
| CSS 类名 | `kebab-case` BEM | `.route-card__title` |

### 函数与文件上限（始终生效）
- 函数 < 50 行 · 文件 < 800 行（典型 200-400）· 嵌套 ≤ 4 层
- 超过 4 层嵌套 → 用 early return 展平

### 安全底线（始终生效）
- 绝不硬编码密钥/Token — 用 `import.meta.env.VITE_xxx`
- 错误消息不泄露内部信息
- 所有外部数据视为不可信，需验证/净化
- 禁止 `console.log` 进入生产代码

### 开发输出规范（始终生效）
写代码前必须先说明：**本次目标 · 所属端 · 涉及文件 · 预期效果 · 验证方式**
